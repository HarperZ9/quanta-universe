Gaussian Splat LOD: Hierarchical Level-of-Detail for 3D Gaussian Splatting

Harper Research Division

Harper Engine Technical Whitepaper HE-WP-010

Abstract

We present Gaussian Splat LOD (GSLOD), a hierarchical level-of-detail system for 3D Gaussian Splatting that enables real-time rendering of massive Gaussian scenes. Through screen-space error-driven LOD selection and smooth blending, GSLOD achieves 10× scalability while maintaining visual quality.

1. Introduction

3D Gaussian Splatting (3DGS) has emerged as a compelling alternative to neural radiance fields, offering real-time rendering of photorealistic scenes. However, 3DGS faces scalability challenges: a typical scene requires millions of Gaussians, each consuming 62 bytes of storage and requiring per-frame sorting and rasterization.

For large-scale scenes—city blocks, architectural complexes, open worlds—naive 3DGS becomes impractical. The O(N log N) sort cost and O(N) rasterization cost scale poorly with Gaussian count.

We introduce Gaussian Splat LOD (GSLOD), a hierarchical level-of-detail system for 3DGS. Key contributions:

1. Hierarchical Gaussian clustering: Parent Gaussians approximate groups of children via ellipsoid fitting
2. Screen-space error metric: LOD selection based on projected approximation error
3. Smooth blending: Geomorphing between LOD levels eliminates popping
4. Streaming integration: Out-of-core rendering of arbitrarily large scenes

GSLOD achieves 10× scalability with minimal visual degradation, enabling 3DGS for production-scale environments.

2. Hierarchy Construction

2.1 Bottom-Up Clustering

Given N Gaussians at LOD 0 (full detail), we construct a hierarchy through iterative clustering:

for level in range(1, max_levels):
    clusters = k_means(gaussians[level-1], k=N[level-1]/8)
    for cluster in clusters:
        parent = fit_ellipsoid(cluster)
        gaussians[level].append(parent)

Each level reduces Gaussian count by approximately 8×.

2.2 Ellipsoid Fitting

A parent Gaussian must approximate its children's appearance. We fit the parent's covariance matrix to minimize reconstruction error:

Σ_parent = argmin_Σ Σᵢ ||render(G_child_i) - render(G(μ, Σ))||²

In practice, we use the covariance of child positions plus average child covariance:

μ_parent = (Σᵢ wᵢ · μᵢ) / Σᵢ wᵢ
Σ_parent = cov(μ_children) + avg(Σ_children)

where wᵢ = opacity_i × volume_i weights by importance.

2.3 Color Aggregation

Parent color (spherical harmonics) is opacity-weighted average:

SH_parent = (Σᵢ opacity_i · SH_i) / Σᵢ opacity_i

This preserves overall appearance while losing high-frequency detail.

2.4 Error Metric Storage

Each parent stores its approximation error:

error = max_i ||render(children) - render(parent)||_FLIP

This error guides runtime LOD selection.

3. Runtime LOD Selection

3.1 Screen-Space Error

For each Gaussian, we compute screen-space error:

sse(G) = G.error × projected_area(G) / pixel_area

where projected_area is the Gaussian's screen footprint. Larger Gaussians contribute more pixels, so their errors are weighted higher.

3.2 LOD Selection

Traversing the hierarchy top-down:

def select_lod(node, threshold):
    if node.sse < threshold:
        return [node]  # Use this node
    else:
        return concat([select_lod(child, threshold) for child in node.children])

The threshold parameter (typically 0.5-2 pixels) controls quality/performance tradeoff.

3.3 View-Dependent Selection

Selection is performed per-frame as view changes. Visible set is determined by:
1. Frustum culling at hierarchy nodes
2. Occlusion culling via Hi-Z test
3. SSE-based LOD selection within visible set

3.4 Smooth Blending (Geomorphing)

To eliminate LOD popping, we blend between adjacent levels:

blend_factor = smoothstep(sse_low, sse_high, sse)
rendered = lerp(render(children), render(parent), blend_factor)

During the transition zone, both parent and children are rendered with complementary opacities.

4. Streaming Architecture

4.1 Tile-Based Organization

The scene is spatially partitioned into tiles. Each tile contains a complete LOD hierarchy for its region. Tile size balances granularity (smaller = finer streaming control) against overhead (larger = fewer tiles to manage).

4.2 Streaming Priority

Tiles are prioritized for streaming based on:

priority(tile) = visibility × screen_coverage × (1 / distance) × direction_weight

where direction_weight favors tiles in the camera's movement direction.

4.3 Memory Budget

A fixed memory budget (e.g., 512MB) limits resident Gaussians. When over budget:
1. Identify lowest-priority tiles
2. Unload fine LODs, retain coarse
3. Continue until under budget

4.4 Prefetching

Based on camera velocity, we predict future tile needs:

predicted_position = camera.position + camera.velocity × lookahead_time
prefetch_tiles_in_frustum(predicted_position)

5.1 Scalability

GSLOD enables rendering of scenes 100× larger than baseline.

5.2 Quality

- Resident with streaming: 512MB (target)

Tile load time: 15ms average (NVMe SSD)
Prefetch hit rate: 94% (no visible stalls)

6. Conclusion

Gaussian Splat LOD transforms 3D Gaussian Splatting from a single-scene technique to a scalable representation suitable for large-scale environments. The hierarchical structure, screen-space error metric, and streaming architecture together enable rendering of arbitrarily large scenes within fixed performance and memory budgets.

The approach is compatible with existing 3DGS content—hierarchy construction is an offline preprocessing step that preserves the original representation as the finest LOD level. Future work includes learned hierarchy construction for improved approximation quality, integration with neural compression for reduced storage, and dynamic LOD for animated Gaussian scenes.
