Cone-Traced Reflections: Roughness-Aware Visibility Gathering for Real-Time Specular

Harper Research Division

Advanced Rendering Technology Group

Technical Whitepaper HE-WP-052

Abstract

We introduce Cone-Traced Reflections (CTR), replacing single-ray reflection queries with visibility cone gathering that naturally handles surface roughness. CTR traces conical beams whose spread matches the specular lobe, collecting all intersected geometry weighted by solid angle contribution. This eliminates the stochastic noise inherent in multi-sample reflection approaches while providing physically correct roughness response. Performance cost scales with cone footprint rather than sample count, achieving 0.8ms for production-quality rough reflections at 1440p.

1. Introduction

Real-time reflections remain challenging because accurate specular requires sampling the hemisphere according to the BRDF. For mirror-like surfaces (roughness ≈ 0), a single reflection ray suffices. As roughness increases, the specular lobe widens, requiring many samples to avoid noise.

Current approaches fall into two categories:
1. Stochastic sampling: Trace multiple rays per pixel, denoise result
2. Screen-space approximation: SSR with fallback to probes

Both have fundamental limitations. Stochastic methods require denoising that blurs detail and introduces temporal artifacts. Screen-space methods fail for off-screen content and struggle with roughness variation.

We propose Cone-Traced Reflections (CTR), which traces visibility cones rather than rays. The cone angle derives directly from surface roughness—wider cones for rough surfaces, narrow cones for smooth. All geometry within the cone contributes, weighted by angular distance from the cone axis.

Key insight: A roughness-matched cone captures the same angular distribution as stochastic sampling, but deterministically. The result is noise-free reflections without denoising, with cost proportional to geometric complexity within the cone rather than sample count.

2. Cone Representation

2.1 Cone Geometry

A visibility cone is defined by:
- Origin o: Reflection point on surface
- Direction d: Primary reflection direction (reflect(-V, N))
- Half-angle θ: Cone spread, derived from roughness

The cone surface at distance t is a disk of radius r(t) = t × tan(θ).

2.2 Roughness to Cone Angle Mapping

For GGX microfacet distribution with roughness α, the specular lobe half-width containing 90% of energy is:

θ = atan(α × k)

where k ≈ 2.0 empirically captures the GGX lobe shape. For α = 0.3 (moderately rough), θ ≈ 31°.

2.3 Energy Distribution Within Cone

Geometry at angular offset φ from cone axis receives weight:

w(φ) = D_GGX(φ) × cos(φ)

where D_GGX is the GGX normal distribution. In practice, we use a simplified Gaussian approximation:

w(φ) = exp(-φ² / (2 × σ²))

with σ = θ / 2.5, ensuring 95% weight within the cone boundary.

3. Cone-Geometry Intersection

3.1 Cone-Triangle Intersection

For each triangle potentially within the cone, we compute:

1. Ray-plane intersection at triangle's plane
2. Closest point on triangle to intersection
3. Angular distance from cone axis
4. Weight based on angular distance

Algorithm:

float3 ConeTriangleContribution(Cone cone, Triangle tri) {
    // Intersect ray with triangle plane
    float t = RayPlaneIntersect(cone.origin, cone.dir, tri.plane);
    if (t < 0) return 0;  // Behind origin
    
    float3 hitPoint = cone.origin + cone.dir * t;
    float3 closestOnTri = ClosestPointOnTriangle(hitPoint, tri);
    
    // Angular distance from cone axis
    float3 toHit = normalize(closestOnTri - cone.origin);
    float cosAngle = dot(toHit, cone.dir);
    float angle = acos(cosAngle);
    
    // Check within cone
    if (angle > cone.halfAngle) return 0;
    
    // Compute weight
    float weight = GaussianWeight(angle, cone.sigma);
    
    // Fetch surface radiance
    float3 radiance = SampleTriangleRadiance(tri, closestOnTri);
    
    return radiance * weight;
}

3.2 Hierarchical Culling

For efficiency, we cull the BVH using cone-box intersection:

bool ConeIntersectsBox(Cone cone, AABB box) {
    // Conservative test: sphere containing cone frustum
    float maxT = cone.maxDistance;
    float maxRadius = maxT * tan(cone.halfAngle);
    
    Sphere boundingSphere;
    boundingSphere.center = cone.origin + cone.dir * (maxT / 2);
    boundingSphere.radius = length(float3(maxRadius, maxRadius, maxT / 2));
    
    return SphereIntersectsBox(boundingSphere, box);
}

Tighter bounds use cone-box separating axis tests, reducing false positives by ~40%.

3.3 Solid Angle Integration

The final reflection gathers all contributions weighted by solid angle:

float3 result = 0;
float totalWeight = 0;

for each triangle in cone:
    float3 contrib = ConeTriangleContribution(cone, tri);
    float solidAngle = TriangleSolidAngle(tri, cone.origin);
    result += contrib * solidAngle;
    totalWeight += solidAngle * GaussianWeight(...);

return result / totalWeight;

4. Implementation

4.1 GPU Architecture

CTR executes as a compute shader post-process on the G-buffer:

[numthreads(8, 8, 1)]
void ConeTraceReflections(uint2 tid : SV_DispatchThreadID) {
    // Load G-buffer
    float3 N = LoadNormal(tid);
    float3 P = LoadWorldPos(tid);
    float3 V = normalize(g_CameraPos - P);
    float roughness = LoadRoughness(tid);
    
    // Skip near-mirror (use ray tracing) and very rough (use probes)
    if (roughness < 0.05 || roughness > 0.7) {
        return;  // Fallback paths handle these
    }
    
    // Construct cone
    Cone cone;
    cone.origin = P + N * 0.01;  // Bias
    cone.dir = reflect(-V, N);
    cone.halfAngle = RoughnessToAngle(roughness);
    cone.maxDistance = g_MaxReflectionDist;
    
    // Trace cone through scene BVH
    float3 reflection = TraceCone(cone);
    
    // Blend with fallback (probes for distant/missing)
    float3 fallback = SampleProbes(P, cone.dir, roughness);
    float confidence = ComputeConeConfidence(cone);
    
    g_Output[tid] = lerp(fallback, reflection, confidence);
}

4.2 BVH Traversal Modification

Standard BVH traversal tests ray-box intersection. We modify to test cone-box:

while (stackPtr > 0) {
    uint nodeIdx = stack[--stackPtr];
    BVHNode node = g_BVH[nodeIdx];
    
    if (!ConeIntersectsBox(cone, node.bounds))
        continue;
    
    if (node.isLeaf) {
        for each triangle in node:
            AccumulateContribution(cone, triangle);
    } else {
        stack[stackPtr++] = node.left;
        stack[stackPtr++] = node.right;
    }
}

4.3 Radiance Caching

Computing shading at every intersected triangle is expensive. We use a radiance cache:

1. Render scene to low-res radiance buffer (1/4 resolution)
2. During cone trace, sample from radiance buffer at hit locations
3. For cache misses, compute shading directly

This reduces per-triangle shading cost by 10×.

4.4 Temporal Stability

Cone tracing is deterministic—same inputs produce same outputs. Unlike stochastic methods, no temporal accumulation or denoising is required. However, we apply temporal filtering for:

- Camera jitter: Subpixel variations cause reflection wobble
- Geometry motion: Moving objects shift intersected geometry

Standard TAA integration handles both cases.

5. Results

5.1 Performance

Measured on RTX 4080 at 2560×1440:

Roughness Range | Triangles/Cone | Time (ms) | vs. 8-sample RT
----------------|----------------|-----------|----------------
0.1 - 0.2       | ~50            | 0.4       | 0.7× (faster)
0.2 - 0.4       | ~200           | 0.6       | 0.5× (faster)
0.4 - 0.6       | ~800           | 0.8       | 0.4× (faster)

CTR is faster than multi-sample ray tracing for rough surfaces because:
1. Single BVH traversal vs. N traversals
2. Coherent memory access (spatial locality within cone)
3. No denoising pass required

5.2 Quality Comparison

We compare against reference (1024 spp path traced reflections):

Method                  | PSNR (dB) | SSIM  | Temporal Stability
------------------------|-----------|-------|-------------------
CTR (ours)             | 38.2      | 0.967 | 0.994
8-sample RT + Denoise  | 35.4      | 0.942 | 0.971
SSR + Probes           | 31.2      | 0.891 | 0.988

CTR achieves higher quality than denoised stochastic methods due to avoiding denoiser artifacts.

5.3 Roughness Response

Visual inspection confirms physically correct roughness behavior:
- α = 0.1: Tight, focused reflections
- α = 0.3: Moderate blur, detail preserved
- α = 0.5: Wide blur, major features only
- α = 0.7: Near-diffuse response

5.4 Failure Cases

CTR degrades when:
- Cone contains thousands of triangles (performance)
- Very glossy surfaces with high-frequency environment (use RT)
- Transparent/refractive geometry (cones don't refract naturally)

6. Conclusion

Cone-Traced Reflections provide a deterministic, noise-free alternative to stochastic reflection sampling. By tracing visibility cones matched to surface roughness, CTR captures the correct angular distribution without multiple samples or denoising.

The technique excels for moderate roughness (0.1-0.6) where stochastic methods require many samples yet single rays are insufficient. Performance scales with geometric complexity rather than sample count, enabling production-quality rough reflections within real-time budgets.

CTR complements existing techniques: ray tracing for mirrors, probes for diffuse, CTR for the challenging middle ground. Together, they form a complete specular solution for modern game engines.
