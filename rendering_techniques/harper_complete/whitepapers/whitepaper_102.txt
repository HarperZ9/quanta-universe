Stochastic Light Trees: O(log N) Importance Sampling for Million-Light Scenes

Harper Research Division

Advanced Rendering Technology Group

Technical Whitepaper HE-WP-102

Abstract

We present Stochastic Light Trees (SLT), a hierarchical data structure enabling O(log N) importance sampling from millions of light sources. Lights are organized into a binary tree where each node stores aggregate luminance and spatial bounds; traversal samples children proportionally to estimated contribution. Combined with ReSTIR temporal reuse, SLT achieves noise-free direct lighting from 1M+ lights at 0.8ms per frame, enabling fully dynamic lighting in large-scale environments.

1. Introduction

Real-time rendering has traditionally limited light counts to hundreds due to per-light evaluation cost. Even with deferred shading and tile-based culling, each visible light requires shadow map lookups and BRDF evaluation—costs that scale linearly with light count.

Modern game environments demand orders of magnitude more lights: every emissive surface, every particle, every window contributes illumination. Offline renderers handle millions of lights via importance sampling—selecting a small subset likely to contribute significantly.

We adapt importance sampling to real-time via Stochastic Light Trees. Lights are organized hierarchically by position and intensity; tree traversal probabilistically descends toward high-contribution lights. The result: evaluating O(log N) lights to sample one, rather than O(N) for exhaustive search.

Combined with ReSTIR's temporal reuse, SLT transforms the many-light problem from intractable to trivial: million-light scenes render at interactive rates with quality matching offline renderers.

2. Tree Construction

2.1 Light Representation

Each light l stores:
- Position p (float3)
- Luminous intensity I (float, candelas)
- Bounding radius r (float)
- Type: point, spot, area (uint2 for parameters)

2.2 Tree Structure

Binary tree with N leaves (one per light) and N-1 internal nodes:

struct LightTreeNode {
    AABB bounds;           // Spatial bounds of all lights in subtree
    float totalIntensity;  // Sum of intensities in subtree
    uint leftChild;        // Index of left child (0 if leaf)
    uint rightChild;       // Index of right child (0 if leaf)
    uint lightIndex;       // Light index if leaf, -1 otherwise
};

2.3 Construction Algorithm

Build tree bottom-up using Morton code sorting:

1. Compute Morton code for each light position
2. Sort lights by Morton code (spatially coherent ordering)
3. Build binary radix tree over sorted lights
4. Propagate bounds and intensity up tree

Construction runs on GPU in O(N log N) time, enabling per-frame rebuild for fully dynamic lights.

3. Importance Sampling

3.1 Contribution Estimation

For shading point x with normal n, estimate light l contribution:

C(l, x, n) = I_l × max(0, dot(n, normalize(p_l - x))) / distance²(x, p_l)

For internal nodes, use conservative upper bound:

C_upper(node, x, n) = node.totalIntensity / min_distance²(x, node.bounds)

3.2 Traversal Algorithm

uint SampleLightTree(float3 x, float3 n, inout uint rngState) {
    uint node = 0; // Root
    
    while (!IsLeaf(node)) {
        uint left = g_Tree[node].leftChild;
        uint right = g_Tree[node].rightChild;
        
        float cLeft = ContributionUpperBound(left, x, n);
        float cRight = ContributionUpperBound(right, x, n);
        float pLeft = cLeft / (cLeft + cRight);
        
        float r = RandomFloat(rngState);
        if (r < pLeft) {
            node = left;
        } else {
            node = right;
        }
    }
    
    return g_Tree[node].lightIndex;
}

3.3 PDF Computation

Sampling PDF is product of probabilities along path:

float ComputePDF(uint lightIndex, float3 x, float3 n) {
    float pdf = 1.0;
    uint node = FindLeaf(lightIndex);
    
    while (node != ROOT) {
        uint parent = g_Tree[node].parent;
        uint sibling = GetSibling(node, parent);
        
        float cNode = ContributionUpperBound(node, x, n);
        float cSibling = ContributionUpperBound(sibling, x, n);
        pdf *= cNode / (cNode + cSibling);
        
        node = parent;
    }
    
    return pdf;
}

4. Integration with ReSTIR

4.1 Initial Candidate Generation

SLT generates initial candidates for ReSTIR reservoirs:

Reservoir InitialSampling(float3 x, float3 n) {
    Reservoir r = {};
    
    for (int i = 0; i < M; i++) { // M = 32 initial candidates
        uint lightIdx = SampleLightTree(x, n, rngState);
        float pdf = ComputePDF(lightIdx, x, n);
        float targetPdf = EvaluateLight(lightIdx, x, n);
        
        float weight = targetPdf / pdf;
        r.Update(lightIdx, weight, rngState);
    }
    
    return r;
}

4.2 Temporal Reuse

Reservoirs carry selected lights across frames:

Reservoir TemporalReuse(float2 uv, Reservoir current) {
    float2 motionVector = g_MotionVectors[uv];
    float2 prevUV = uv - motionVector;
    
    Reservoir history = g_PrevReservoirs[prevUV];
    
    // Validate history sample still visible
    if (ValidateSample(history, uv)) {
        current.Merge(history, 20); // Cap history at 20 frames
    }
    
    return current;
}

4.3 Spatial Reuse

Share samples with neighbors for variance reduction:

Reservoir SpatialReuse(float2 uv, Reservoir r) {
    for (int i = 0; i < 5; i++) {
        float2 offset = PoissonDisk[i] * RADIUS;
        float2 neighborUV = uv + offset;
        
        Reservoir neighbor = g_Reservoirs[neighborUV];
        
        if (SimilarSurface(uv, neighborUV)) {
            r.Merge(neighbor, 1);
        }
    }
    return r;
}

5. Results

5.1 Performance Scaling

Time (ms) vs light count at 1440p:

Lights    | Naive    | Tiled    | SLT+ReSTIR
----------|----------|----------|------------
1K        | 0.8      | 0.3      | 0.2
10K       | 8.2      | 2.1      | 0.3
100K      | 82       | 18       | 0.5
1M        | timeout  | 180      | 0.8

SLT scales logarithmically; 1M lights cost only 4× more than 1K.

Tree nodes: 32 bytes each
1M lights → 64MB tree storage
Reservoirs: 16 bytes/pixel → 7MB at 1440p

6. Conclusion

Stochastic Light Trees transform many-light rendering from a scalability problem to a solved problem. By importance sampling lights proportional to estimated contribution, SLT reduces per-frame light evaluation from O(N) to O(log N).

Combined with ReSTIR, the approach delivers production-quality results from million-light scenes at real-time rates. This enables new categories of content: fully emissive environments, city-scale night scenes, and particle-based volumetric lighting all become practical.

The technique integrates with existing engines via drop-in replacement of light loop—no fundamental rendering changes required. We expect SLT to become standard practice as content creators demand ever-more-detailed lighting.
